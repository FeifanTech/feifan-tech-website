规划管理 Prompt 模板库
对应模块二：项目规划与进度管理
🗓️ 里程碑计划生成 场景：项目启动初期 展开 ▼
输入项目信息，生成带关键节点的里程碑计划表，包含依赖关系和风险提示。
你是一名资深项目经理，请为以下项目生成里程碑计划： 项目名称：[填写] 项目类型：[如：SaaS 研发 / 系统集成 / 电商平台] 团队规模：[前端/后端/测试/设计 各 X 人] 总工期：[X 个月] 关键约束：[如：固定上线日期 / 资源限制] 主要功能模块：[列举 4-6 个] 请生成： 1. 完整里程碑计划表（里程碑名称 / 目标完成日期 / 成功标准 / 依赖条件） 2. 关键路径说明 3. 缓冲期建议（哪些阶段需要留 buffer） 4. 各里程碑的交付物清单
💰 工作量三点估算 场景：需求评估阶段 展开 ▼
描述功能需求后，AI 给出乐观/最可能/悲观三点估算，并计算期望工作量。
你是一名有 10 年经验的技术项目经理，精通工作量估算。 请对以下功能进行三点估算（PERT 方法）： 功能描述：[填写功能详细描述] 技术栈：[填写] 团队能力：[初级/中级/高级工程师，几人] 类似历史项目：[如有，描述一下] 请输出： 1. 乐观估算（O）：一切顺利时 2. 最可能估算（M）：正常情况 3. 悲观估算（P）：遇到典型风险 4. PERT 期望值：(O + 4M + P) / 6 5. 主要不确定性因素（影响估算准确性的关键点） 6. 提高估算准确性的建议（需要澄清的问题）
📝 周报/月报自动生成 场景：日常进度汇报 展开 ▼
将工作流水账丢给 AI，5 分钟内生成结构化周报或月报。
请根据以下工作记录，生成一份专业的项目[周报/月报]： 时间范围：[X 月 X 日 - X 月 X 日] 原始记录（可以是流水账）： [直接粘贴你的日报、聊天记录摘要等] 要求： 1. 格式：完成事项 / 下周计划 / 风险与问题 / 关键指标 2. 语言精炼，每项不超过 2 句话 3. 突出成果（结果导向），而非过程 4. 风险部分：明确需要什么支持 5. 关键指标：进度完成率 / 预算使用率 6. 总字数控制在 500 字以内
🔄 变更申请文档生成 场景：需求变更管理 展开 ▼
描述变更需求，AI 生成包含影响分析的标准变更申请文档。
请生成一份变更控制申请文档： 变更描述：[填写需要变更的内容] 变更原因：[业务原因/技术原因] 变更类型：[需求变更/技术变更/资源变更] 当前项目状态：[阶段 + 进度] 请分析并生成： 1. 变更内容详细描述 2. 影响范围评估（时间线 / 资源 / 预算 / 其他需求） 3. 风险分析（实施变更 vs 不实施变更的风险） 4. 实施方案（步骤 + 工期估算） 5. 建议决策（推荐/不推荐，理由） 6. 签批清单（需要谁审批）

协同管理 Prompt 模板库
对应模块三：AI 赋能产研协同
📋 需求澄清问题清单 场景：需求评审前 展开 ▼
将需求文档丢给 AI，生成需要向业务方澄清的问题清单，减少评审会上的无效讨论。
你是一名资深产品经理，擅长发现需求中的模糊和遗漏。 请分析以下需求描述，列出： 1. 5-10 个需要向业务方澄清的问题（按优先级排序） 2. 可能存在的边界情况和异常场景 3. 被遗漏的非功能性需求（性能/安全/兼容性） 4. 与其他功能可能存在的依赖关系 5. 建议追加的验收标准 需求描述：[粘贴需求原文]
🎯 Epic 拆分为 User Story 场景：Sprint 规划前 展开 ▼
将大型 Epic 自动拆分为多个独立的 User Story，并推荐实施优先级。
你是一名经验丰富的产品经理，精通敏捷开发和 User Story 写作。 请将以下 Epic 拆分为可以在一个 Sprint 内完成的 User Story： Epic 描述：[填写 Epic 名称和描述] 业务目标：[填写此 Epic 要实现的业务价值] 技术约束：[如有，描述主要约束] 请为每个 User Story 提供： 1. 故事格式：作为[角色]，我希望[功能]，以便[价值] 2. 验收标准（AC）：5-6 条，具体可测试 3. 故事点估算（1/2/3/5/8） 4. 实施优先级（P1/P2/P3） 5. 依赖关系（是否依赖其他 Story） 总体原则：每个 Story ≤ 8 点，可在 1 个 Sprint 内独立交付价值
📧 技术阻塞上报邮件 场景：项目遇到技术阻塞 展开 ▼
描述阻塞问题，AI 生成结构清晰、明确说明需要什么支持的上报邮件。
请帮我起草一封技术阻塞问题上报邮件： 问题描述：[填写技术阻塞的具体情况] 已尝试的解决方案：[填写团队已经尝试过的方法] 影响：[填写对项目时间线/功能的影响] 需要的支持：[填写需要什么人/资源来解决] 紧急程度：[低/中/高，原因] 邮件要求： 1. 开门见山，第一段说明问题和影响 2. 清楚描述已做的努力，避免让人觉得没有自救 3. 明确提出需要的支持（具体到人/时间/资源） 4. 给出 2 个可能的解决方向（供讨论） 5. 语气专业，不制造恐慌

客户沟通 Prompt 模板库
对应模块四：AI 强化客户沟通
⚠️ 坏消息传达邮件 场景：项目延期/超预算/质量问题 展开 ▼
传达延期、超预算等坏消息时，AI 帮你结构化表达，解决方案前置，降低客户负面情绪。
请帮我起草一封传达坏消息的客户邮件： 坏消息类型：[延期/超预算/质量问题/范围变更] 具体情况：[描述什么发生了变化，变化多大] 原因：[客观原因 + 团队因素] 已有的解决方案：[我们打算怎么解决] 对客户的影响：[影响哪些方面，有多严重] 客户特点：[描述客户的关注点和沟通风格] 邮件结构要求： 1. 开门见山（第一段直接说明情况，不绕弯子） 2. 承担责任（不推卸，即使有客观原因） 3. 解决方案前置（方案比问题占更多篇幅） 4. 明确下一步行动（谁/做什么/何时） 5. 结尾积极（表达信心，不失去客户信任） 6. 控制在 300 字以内
🔄 变更请求拒绝/协商 场景：客户提出超范围需求 展开 ▼
礼貌但清晰地拒绝或协商超范围变更请求，维护项目边界同时保持良好关系。
请帮我起草一封回应客户变更请求的邮件： 客户变更请求：[描述客户提出了什么] 此变更不在范围内的原因：[合同/时间/资源约束] 变更影响评估：[需要增加多少工期/成本] 我们的立场：[完全拒绝/需要走变更流程/有条件接受] 邮件要求： 1. 先认可客户的需求合理性（同理心） 2. 客观说明此变更的影响（不带情绪） 3. 提供 2 个选项（例：走变更流程 or 放到下一期） 4. 清楚说明我们能做什么（而非强调不能做什么） 5. 不让客户感觉被拒绝，而是感觉被尊重和专业对待
🎉 项目价值总结报告 场景：项目收尾/续约准备 展开 ▼
项目收尾时，AI 生成突出业务价值的项目总结报告，为续约和口碑传播奠定基础。
请生成一份突出业务价值的项目总结报告： 项目基本信息：[名称/周期/团队规模] 主要交付成果：[功能列表] 量化数据：[用户数/处理量/性能指标/成本节省等] 克服的主要挑战：[列举 2-3 个] 客户反馈亮点：[如有正面反馈] 报告结构： 1. 执行摘要（3 句话，面向决策层） 2. 交付价值概览（量化 ROI） 3. 里程碑完成情况（可视化时间线） 4. 技术亮点（差异化能力说明） 5. 客户反馈（引用/总结正面评价） 6. 未来合作建议（自然引出续约机会）

风险管理 Prompt 模板库
对应模块五：AI 风险识别与消除
🔍 项目启动风险全扫描 场景：项目启动第一周 展开 ▼
项目启动时，从 5 个维度对项目进行系统化风险扫描，建立初始风险登记册。
你是一名风险管理专家，请对以下项目进行全面的风险扫描： [填写项目名称/类型/规模/周期/技术栈/团队/客户/约束] 请从以下 5 个维度识别风险，每个维度 3-5 个： 1. 📋 范围与需求风险 2. 🔧 技术与架构风险 3. 👥 资源与团队风险 4. 🤝 干系人与沟通风险 5. ⚙️ 外部依赖与环境风险 每个风险包含： - 风险描述（一句话） - 可能性（高/中/低） - 影响程度（高/中/低） - 风险等级 = 可能性 × 影响 - 早期预警信号（怎么判断正在发生） - 建议的预防措施（1-2 条） 最后：按风险等级排序，输出 TOP5 高优先级风险
🚨 危机 24 小时应对方案 场景：项目出现紧急情况 展开 ▼
项目发生危机时，AI 快速生成未来 24 小时的行动清单和客户沟通方案。
项目发生了以下紧急情况，需要立即处理： 情况描述：[填写发生了什么] 影响范围：[填写受影响的功能/用户/数据/时间线] 发现时间：[填写] 已采取的措施：[填写已经做了什么] 请立即提供： 1. 快速影响评估（时间线/质量/客户信任/商务风险） 2. 未来 24 小时行动清单（按优先级，含责任人） 3. 客户/上级通报邮件草稿（让对方知道我们在控制局面） 4. 可能的解决方案（3 个选项，含时间/代价分析） 5. 危机稳定后的复盘要点

质量交付 Prompt 模板库
对应模块六：AI 交付质量管控
🧪 UAT 验收清单生成 场景：UAT 准备阶段 展开 ▼
基于功能描述自动生成 UAT 验收清单，覆盖正常场景、边界场景和异常场景。
请为以下功能设计完整的 UAT 验收清单： 功能名称：[填写] 功能描述：[填写功能的详细说明] 用户角色：[填写使用此功能的用户类型] 业务规则：[填写关键的业务规则和限制] 已知边界条件：[如有，填写] 请生成： 1. 功能验收清单（正常路径 + 替代路径 + 异常处理） 2. 非功能验收清单（性能/安全/兼容性/无障碍） 3. 数据验证清单（边界值/特殊字符/空值处理） 4. 硬性验收标准（上线前必须通过的 Must-Pass 项） 5. 探索性测试建议（自由测试时重点关注的区域）
🚀 上线发布保障方案 场景：系统上线前 展开 ▼
为系统上线生成完整的发布保障方案，包括检查清单、执行步骤和回滚计划。
请为以下系统生成完整的上线发布保障方案： 系统描述：[填写系统名称和架构概述] 上线时间：[填写计划上线时间] 影响范围：[填写影响的用户/业务] 主要技术风险：[列举已知风险点] 请生成： 1. 上线前检查清单（技术 20 条 + 业务 10 条） 2. 上线执行步骤（按时序，含负责人和预计时间） 3. 监控指标清单（上线后前 2 小时重点监控的指标） 4. 回滚触发条件（什么情况下需要立即回滚） 5. 回滚步骤（技术回滚 + 客户通知模板） 6. 上线公告模板（对用户发布的版本更新说明）

AI 工具选型指南
项目交付场景的推荐工具及最佳用途
🤖
Claude
首选推荐
擅长长文档分析、复杂推理和结构化写作，特别适合项目文档生产和策略分析。上下文窗口大，可以处理完整的项目文档。
最佳使用场景
项目章程、周报、月报生成
风险分析和应对预案
长邮件/汇报材料起草
需求文档分析和澄清
💬
ChatGPT / GPT-4o
通用推荐
通用能力强，插件生态丰富，代码能力出色。适合需要快速迭代和多模态（图片/表格）处理的场景。
最佳使用场景
数据分析（Advanced Data Analysis）
表格/图表解读
快速 Q&A 和头脑风暴
代码审查和问题排查
🎙️
飞书妙记
会议首选
中文识别准确率高，与飞书生态深度集成。会议结束后自动生成转录和摘要，支持一键生成会议纪要。
最佳使用场景
客户会议实时转录
内部周会自动纪要
面试/访谈记录整理
🎨
Gamma
汇报利器
AI 驱动的演示文稿生成工具，输入大纲或文字，10 分钟生成专业的项目汇报 PPT。
最佳使用场景
客户阶段汇报 PPT
项目启动说明材料
方案对比演示
🔄
Dify
自动化
开源 AI 工作流平台，可以将多个 AI 步骤串联成自动化流程。适合团队共享的固定流程自动化。
最佳使用场景
周报自动收集和生成
项目文档知识库问答
风险扫描自动化工作流
📋
Notion AI
知识管理
将 AI 能力直接嵌入项目文档工作区，支持内容生成、摘要、翻译和问答，适合项目知识库建设。
最佳使用场景
项目知识库搭建和问答
经验总结和最佳实践沉淀
跨项目信息查询

持续学习路径
课程结束后，如何保持 AI 能力持续进化

📅 30 天行动计划

第 1-10 天
• 建立个人 Prompt 库
• 每天用 AI 完成 1 项工作
• 记录时间节省数据
• 完成周报自动化
第 11-20 天
• 在新项目用 AI 做完整启动
• 用 AI 做一次风险扫描
• 建立 User Story 模板库
• 团队分享 1 个成功案例
第 21-30 天
• 搭建第一个 Dify 工作流
• 与团队共建 Prompt 库
• 总结量化收益数据
• 制定下季度提升计划
💡
记住：AI 能力没有终点线。最重要的是建立持续实验的习惯——每周尝试一个新场景，每月回顾一次 Prompt 库，每季度与团队分享最佳实践。
