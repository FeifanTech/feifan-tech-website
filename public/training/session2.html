模块 1｜提示词设计的底层逻辑
20 分钟 · 认知建立
工程师视角的类比：提示词 = 函数签名
💡
核心类比：提示词就是函数签名

一个好的提示词和一个好的函数一样，需要明确输入、约束、输出格式和边界情况。模糊的函数签名会产生不可预期的行为，模糊的提示词也一样。

// 函数设计视角 prompt_function()
Input（输入） 背景信息、原始材料、文档、代码片段
Constraints（约束） 限制条件（不超过 500 字）、禁止事项（不要生图）
Output Format（格式） Markdown / JSON / 代码 / 表格 / 要点列表
Edge Cases（边界） 遇到不确定情况时怎么处理（"如果信息不足，请列出需要补充的内容"）
提示词的四个核心要素（详解）

🎭 角色（Role）

告诉 AI 它是谁，决定知识视角和表达风格。角色越具体，输出越专业。

❌ 弱："你是一个程序员"
✅ 强："你是一位有 10 年经验的 Java 架构师，专注于高并发系统设计，熟悉 Spring Cloud 生态"

🌍 背景（Context）

当前情况，越具体越好。包含：项目背景、技术栈、团队规模、约束条件、已有尝试。

背景决定 AI 的"参考系"。同一个问题，背景不同，答案可以完全不同。

📋 任务（Task）

要做什么，用动词开头，避免歧义。一个提示词最好只完成一个清晰的任务。

好的动词：请分析、请生成、请评审、请设计、请重构、请翻译、请提取

📐 格式（Format）

输出成什么样。不指定格式时，AI 会自行决定，往往不是你想要的。

选项：Markdown 表格 / JSON / 代码块 / 有序列表 / 要点摘要 / 指定字数范围

让 AI 扮演特定专家角色，显著提升垂直领域输出质量

请从架构安全性角度审查以下代码，重点关注线程安全和资源泄漏。

少样本示例（Few-shot Prompting）
给 AI 看 1-3 个"这是我想要的风格/格式"，AI 自动模仿

当你需要 AI 保持特定的风格、语气或格式时，直接给示例比文字描述更有效。AI 擅长模式匹配。

以便 [获得某种价值]

  - [ ] 条件1
  - [ ] 条件2

🛠️
实操练习：找一份公司过去写的优质用户故事或需求文档，作为示例让 AI 按相同格式生成新需求。
3
思维链（Chain of Thought，CoT）
加上"请一步一步思考"，让 AI 先推理再给结论

对于复杂问题，让 AI 展示推理过程而不是直接给结论，可以显著降低错误率。AI 在"思考"时能发现自己的逻辑漏洞。

适用场景示例
复杂 Bug 定位"请一步一步分析这个 NullPointerException 的可能根因..."
技术选型分析"请逐步比较 Kafka vs RabbitMQ 在以下场景的适用性..."
性能问题排查"请一步一步分析这段 SQL 的执行计划瓶颈..."
架构风险评估"请逐步推演这个方案在高峰流量下的失效模式..."
 
请一步一步分析这个内存溢出问题的可能原因。
已知信息：
- JVM 堆大小：4G，目前使用 3.8G
- 问题在每天凌晨 2 点左右出现
- GC 日志显示 Full GC 频率异常

步骤：
1. 首先分析可能的内存泄漏类型
2. 根据时间规律推断可能的定时任务或批处理
3. 给出优先级最高的排查路径
4. 最后给出修复建议
4
输出格式控制
要求特定结构输出，便于后续处理或直接复用

要求 AI 输出 JSON / Markdown 表格 / 特定数据结构，可以直接喂给下游系统处理，节省大量手工整理时间。

请将以下需求文档中的功能点提取出来，输出为 JSON 格式。

5
迭代追问（Iterative Refinement）
AI 不是一次对话就结束的，学会用追问持续优化输出

把 AI 交互当作一个迭代精炼的过程，而不是一次问答。每次追问都在让输出向你的目标靠近一步。

📋 追问句式库

深化展开：

"请把第三点展开，给出具体实现步骤"
"这个方案的风险点是什么？"
"给出一个反例，什么情况下这个方案会失效？"

调整方向：

"如果客户预算只有原来的一半，方案如何调整？"
"用更简洁的语言重写，面向非技术的业务方"
"从甲方/乙方/用户的视角重新描述一下"

6
System Prompt 设计
在平台层固化 AI 的角色和行为边界

System Prompt 是对话开始前就设定好的"AI 行为规范"，相当于给 AI 写一份岗位职责书。在 Dify 等平台搭建应用时必用。

👨‍🏫
🛠️ 实操：在 Dify 中创建"需求分析助手"

在 Dify 中新建一个对话应用，将以下 System Prompt 粘贴进去，并测试效果

你是一位专业的需求分析专家，拥有 10 年软件项目交付经验。
你的任务是帮助工程师和交付团队分析客户需求文档。

【行为规范】
1. 始终以结构化格式输出分析结果（功能清单 / 模糊点 / 风险点 / 确认问题）
2. 对每个功能点评估优先级（P0 关键/P1 重要/P2 次要）
3. 主动识别需求中的技术风险，不要遗漏
4. 遇到描述模糊的需求，必须提出确认问题而不是假设
5. 输出格式统一使用 Markdown，便于复制到项目管理系统

【输出格式】
## 1. 功能需求清单
（表格形式，包含：功能名称/优先级/涉及模块/预估复杂度）

## 2. 模糊点识别  
（逐条列出，说明模糊原因）

## 3. 技术风险点
（按严重程度排序）

## 4. 需向客户确认的问题
（可直接发给客户的问题列表）

  • 生成修复方案 + 解释为什么这样修
  • 建议防止类似 Bug 的编码规范

🎯 核心实操演示：AI Code Review 完整流程
1

准备代码（1 分钟）

找一段 50-100 行的真实业务代码（Java/Python/Go 均可），进行必要的脱敏处理（替换真实变量名和业务逻辑描述）。

2

发给 Claude（2 分钟）

使用以下 Prompt 模板：

你是一位有 10 年经验的高级工程师，专注于代码质量和工程实践。
请对以下代码进行全面 Code Review，重点关注：
1. 潜在 Bug 或逻辑错误
2. 性能瓶颈（时间复杂度/空间复杂度/数据库查询）
3. 安全漏洞（SQL 注入/XSS/不安全反序列化等）
4. 可读性和可维护性问题
5. 是否符合 SOLID 原则

输出格式：按严重程度（Critical/Major/Minor）分类，每条问题说明：
- 问题位置（行号或函数名）
- 问题描述
- 修复建议（附修复后的代码片段）
