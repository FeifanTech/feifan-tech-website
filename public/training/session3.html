模块 1｜为什么需要 RAG？
15 分钟 · 认知建立
大模型的核心局限

⚠️ 大模型"不知道你"的问题

大模型的知识截止于训练时，它不知道你的公司、你的产品、你的客户、你的内部文档。

❌ 通用大模型的回答
用户：我们产品的 SLA 条款是什么？
AI：抱歉，我没有关于贵公司产品
    SLA 条款的具体信息。通常 SLA
    包括...（开始瞎编通用内容）
✅ RAG 增强后的回答
用户：我们产品的 SLA 条款是什么？
AI：根据贵公司产品手册（v2.3）：
    - 系统可用性：99.9%/月
    - 故障响应：P0 级 30 分钟内
    - 数据恢复：RPO ≤ 4 小时
    [来源：产品手册第 12 页]
RAG 工作原理直觉理解
📖 RAG（Retrieval-Augmented Generation，检索增强生成）= 大模型 + 实时检索你的私有文档
类比：RAG 相当于给大模型配了一个"随时能查公司内部资料的助理"，现查现答，不靠记忆。

"SLA 是什么？"

→

最相关的 3-5 段

的准确回答

⚠️ 关键：AI 基于检索到的真实文档片段回答，而不是凭训练时的记忆"瞎猜"

RAG 对公司的具体价值
🎯

售前知识库

产品功能说明、历史方案、竞品对比，AI 实时检索回答客户问题，售前响应速度提升 10x。

📦

交付知识库

历史项目文档、标准流程、问题解决方案，新人入职和项目交付不再依赖老员工记忆。

⚙️

技术知识库

内部技术规范、API 文档、架构决策记录（ADR），技术决策有据可查，减少重复踩坑。

模块 2｜用 Dify 搭建 RAG 知识库
35 分钟 · 全程实操，从零到跑通
实操流程（从零到跑通）
1

在 Dify 创建知识库

登录 Dify 平台 → 知识库 → 新建知识库 → 命名（如"产品手册知识库"）→ 上传文档（支持 PDF/Word/TXT/Markdown）。用公司真实脱敏文档演示效果更好。

2

配置分块策略（Chunk Size）

理解分块的重要性：太大的块召回精度低，太小的块信息不完整。

文档类型推荐 Chunk Size原因
技术文档/规范500-800 字符段落结构清晰，自然边界合适
FAQ 列表200-400 字符每条 Q&A 应独立召回
长篇报告800-1200 字符保持完整语义段落
代码文档300-600 字符按函数/类分块
3

创建对话应用，关联知识库

新建对话型应用 → 在上下文配置中关联刚才创建的知识库 → 设置 Top-K 召回数量（建议 3-5）。

4

写 System Prompt（企业级 RAG 最重要配置）

你是公司产品知识助手，专门回答关于产品功能、技术规格和解决方案的问题。

【重要规则】
1. 仅基于提供的知识库内容回答问题
2. 如果知识库中没有相关信息，明确告知用户"知识库中暂无此信息"，不要编造
3. 回答时注明信息来源（文档名称和章节）
4. 如果问题模糊，先询问澄清，再给出回答
5. 对于知识库有的内容，给出准确、简洁的回答，并标注置信度

【输出格式】
- 直接回答问题
- 来源：[文档名称 - 章节/页码]
- 相关延伸：如有关联内容可主动提及
5

测试与对比

测试两类问题：① 知识库里有的问题（应给出有来源的准确回答）；② 知识库里没有的问题（应说"不知道"而不是瞎编）。对比差异，验证 System Prompt 的效果。

常见问题与调优
问题现象根因解决方向
AI 还是在瞎编 System Prompt 没有明确限制 加"仅基于知识库内容回答"约束；增加"不知道就说不知道"指令
检索不准，找错段落 Chunk Size 太大 / 文档质量差 调小分块大小；清洗文档格式（去除冗余表头/页脚）
回答不完整 检索召回数量不足 增加 Top-K 召回数（从 3 增到 5-8）
相似问题答案不一致 文档存在矛盾内容 清理知识库，建立文档版本管理机制
响应速度慢 文档量大 / 嵌入模型慢 选用更快的 Embedding 模型；优化知识库分类
⚠️
企业级 RAG 最重要的一条原则

System Prompt 中必须明确指定 "仅基于知识库内容回答"，否则大模型会用自己的训练知识"补充"，导致回答混入不可靠信息。

模块 3｜Skills 机制——让 AI 具备可扩展的专项能力
30 分钟 · 本期新增核心内容
🎯
Skills 是从"会用 AI"到"设计 AI 能力"的关键跨越

本模块是第三期新增的核心内容。掌握 Skills 机制，让你从 AI 使用者变成 AI 能力设计者。

什么是 Skills？

📌 Skills = 固化的 SOP

Skills（技能）是为 AI 预定义的、结构化的专项能力模块，通过 提示词 + 工具 + 知识 的组合，让 AI 在特定场景下始终以专业、一致的方式执行任务。

🔤 提示词（临时指令）
每次用的时候都要重新输入
质量取决于用的人会不会写
无法保证一致性
不可复用、不可分享
⚙️ Skill（固化的 SOP）
一次配置，团队共享使用
触发即按标准 SOP 执行
质量一致，可持续优化
可复用、可组合、可扩展
Skills 的三个核心要素
🎯

触发条件（Trigger）

什么情况下启用这个技能？
例：收到含"需求"关键词的消息时 / 用户上传文档时 / 被显式调用时

⚙️

执行逻辑（Workflow）

这个技能的处理步骤是什么？
例：提取需求 → 识别模糊点 → 评估风险 → 生成确认清单

📊

输出规范（Output）

产出物的格式和质量标准是什么？
例：Markdown 结构化报告 / JSON / 特定表格格式

Skills 的三种形态
🔤 Prompt Skill

纯提示词组合，无需编码

通过精心设计的 System Prompt 固化 AI 行为，最简单易上手的 Skill 类型。

示例："需求评审专家"：始终检查 6 类问题
适合：有固定模板的重复性任务
🔧 Tool Skill

调用外部工具的技能

AI + 工具调用，可以执行搜索、API 调用、数据库查询等操作。

示例："竞品分析"：联网搜索 + 结构化输出
适合：需要实时数据或外部系统的任务
🔄 Workflow Skill

多步骤自动化技能

把多个 AI 步骤和工具步骤串联成完整的自动化流程。

示例："合同风险扫描"：上传→提取→分析→报告
适合：流程固定的复杂任务
公司级 Skills 建设路线图（优先级推荐）
P0

需求分析 Skill

输入原始需求，输出：结构化用户故事 + 模糊点清单 + 技术风险评估

频率：每个项目必用 | 节省时间：每次 2-3 小时 → 15 分钟
P0

代码 Review Skill

输入代码，输出：问题清单（按 Critical/Major/Minor 排序）+ 改进建议

频率：每次 PR/代码提交 | 节省时间：每次 1 小时 → 5 分钟
P1

售前方案 Skill

输入客户需求，输出：方案大纲 + 差异化竞争角度 + 价值主张

频率：每次售前响应 | 节省时间：每次 4-6 小时 → 1 小时
P1

运维排障 Skill

输入错误日志，输出：根因分析 + 处理步骤 + 预防措施 + SOP

频率：每次线上故障 | 节省时间：排障时间缩短 50%
P2

合同审查 Skill

输入合同文本，输出：关键条款提取 + 风险点标注 + 审查意见

频率：每次合同签署前 | 节省时间：需律师复核，AI 做初筛
🛠️ 实操：在 Dify 中创建"需求分析 Skill"
1

定义触发条件和输入格式

在 Dify 中新建工作流（Workflow）应用，添加"开始节点"，定义输入变量：requirement_doc（需求文档文本）。

2

配置处理步骤（4 个 AI 节点串联）

节点 1：功能提取
 提示：从以下需求文档中提取所有功能点，
       输出 JSON 格式（id/name/priority/module/complexity）
       
节点 2：模糊点识别  
 提示：基于上一步的功能列表，识别描述模糊的功能点，
       说明模糊原因，输出需要确认的具体问题

节点 3：技术风险评估
 提示：评估上述功能列表中的技术实现风险，
       按严重程度（高/中/低）排序，给出规避建议

节点 4：生成确认清单
 提示：整合前三步的分析，生成一份可直接发给
       客户的确认问题清单（简洁、专业）
3

固定输出格式

在最后一个节点设置输出格式：Markdown 结构化报告，包含四个章节。添加"结束节点"，配置展示变量。

4

测试并迭代优化

用一份真实脱敏需求文档测试，检查各节点输出质量。如果某个节点质量不够，单独优化该节点的提示词，直到输出符合预期。

Skills vs RAG vs 工作流的关系
RAG → 给 AI 提供知识 （知道什么，What to know）
Skills → 给 AI 定义能力 （会做什么，What to do）
工作流 → 串联 Skills 完成复杂任务 （怎么做，How to do）
三者结合 = 知识 + 能力 + 流程 = 完整的 AI 应用

🛠️ 实操任务：搭建"需求文档自动分析工作流"
👨‍🏫
讲师说明：约 20 分钟完成以下工作流

使用 Dify Workflow，搭建一个三步 AI 流程，实现需求文档的自动分析。

📋 工作流设计规格

#节点类型输入AI 任务输出格式
0 开始 用户粘贴需求文档 — 文本变量
1 AI 节点 需求文档全文 提取功能列表 JSON（id/name/priority/complexity）
2 AI 节点 功能列表 识别技术风险点 Markdown 要点列表
3 AI 节点 功能列表 + 风险点 生成客户确认问题清单 可直接发给客户的问题列表
4 结束 三个节点的输出 — 综合报告展示
🚀
进阶（选做）：接入飞书 API，把工作流输出自动发送到指定飞书群，实现真正的自动化交付。
