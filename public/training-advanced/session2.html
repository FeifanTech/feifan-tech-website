认知建立⏱ 15 分钟

测试工作的本质是什么？

建立质量信心
测试的核心价值不是"找 Bug"，而是让团队知道哪些地方是安全的，哪些地方是风险点。AI 让这种信心建立更加系统化。

测试用例设计的 AI 革命

💻 实操⏱ 45 分钟

❌ 传统方式的问题
  • 依赖个人经验，不同人覆盖率差异大
  • 容易遗漏边界值和异常场景
  • 用例描述格式不统一
  • 新人需要大量时间积累经验
  • 重复性工作耗费大量精力
✅ AI 辅助的改变
  • 系统性应用等价类/边界值等方法
  • 自动生成完整覆盖矩阵
  • 标准化输出格式（表格/YAML/JSON）
  • 新人使用 AI 也能达到资深测试水平
  • 人专注高价值的探索性测试

技法二：场景法与判定表

复杂业务规则的系统化拆解

输入业务规则（如：折扣规则、权限矩阵、状态流转），让 AI 生成判定表，并从判定表导出最小测试用例集（避免冗余，确保覆盖）。

💬 判定表生成提示词
以下是一个业务规则描述：
[粘贴业务规则，如折扣规则、权限判断、状态转换]

示例：
  用户类型（普通/VIP/企业），购买金额（<100/100-500/>500），
  是否使用优惠券，根据这三个条件决定折扣率。

请：
1. 生成完整的判定表（所有条件组合 × 所有结果）
2. 识别并合并等价的测试用例（条件合并/无关条件）
3. 导出最小测试用例集（去除冗余后）
4. 标出高优先级测试用例（基于风险和业务重要性）

技法三：从 PRD + AC 直接生成测试用例矩阵

PM+QA 兼任角色最高效的使用方式——复用第一期成果

🔗 与第一期的衔接
直接使用第一期产出的 PRD 章节（含 AC），作为本技法的输入。这正是"上一个环节的 AI 输出直接作为下一个环节的 AI 输入"的核心体现。
💬 PRD → 测试用例矩阵（核心提示词）
角色：你是一位同时具备产品和测试思维的高级测试工程师

输入 1 - PRD 功能描述：
[粘贴功能描述]

输入 2 - 验收条件（AC）：
[粘贴 Given-When-Then 格式的 AC]

任务：
  1. 提取所有可测试点（每个 AC 必须对应至少一个测试用例）
  2. 为每个测试点生成测试用例（正常流 + 异常流 + 边界场景）
  3. 识别高风险区域，标注需要重点测试的场景
  4. 指出 PRD 中描述不清晰、无法测试的地方（反向审查需求）

输出格式：
  A. 测试用例表格：用例ID | 模块 | 场景描述 | 前置条件 | 操作步骤 | 预期结果 | 优先级 | 对应AC编号
  B. 风险标注：单独列出，说明风险原因和建议
  C. 需求疑问：单独列出，作为需求评审反馈
💻
🎯 实操任务（20 分钟）
用第一期产出的 PRD 章节（含 AC），直接生成对应的测试用例矩阵。特别注意 AI 输出的 C 部分"需求疑问"——这些往往是需求中真正的问题所在，也是 QA 反向审查需求的核心价值。

技法四：探索性测试方向生成

难以穷举场景的智能化测试策略

对于难以用用例穷举的场景（如 UI 交互、复杂业务流），让 AI 生成探索性测试的 Charter（测试宪章）：

🎯
目标区域
明确本次探索测试聚焦的功能模块和业务流程，避免漫无目的地测试。
🔬
探索策略
AI 建议的具体探索方向，如：快速操作、异常中断、权限切换、多设备等。
⚠️
重点风险类型
基于功能特征推断的高风险场景，如：并发操作、状态一致性、数据完整性。

自动化测试脚本生成

💻 实操（技术重点）⏱ 40 分钟

🔄 AI 如何改变自动化测试的门槛
过去：写自动化脚本需要扎实的编程能力，测试工程师花大量时间在"如何写代码"上。
现在：你只需要描述"测试意图"，AI 生成脚本，你来审查逻辑 + 调试 + 维护。编程能力要求从"能写"变成"能读"。
4a

场景一：Pytest 接口自动化测试

描述意图，AI 生成可运行的 Python 测试代码

💬 Pytest 自动化测试脚本生成
角色：你是一位精通 Python 自动化测试的工程师

接口文档：
  URL: POST /api/v1/users/register
  请求体: {"name": string, "phone": string, "email": string, "password": string}
  响应: {"code": 0, "data": {"userId": string}, "message": "success"}
  错误码: 400(参数错误) / 409(手机号已注册) / 500(服务器错误)

测试场景：
  1. 正常注册（新手机号，完整参数）
  2. 手机号已注册（预期 409）
  3. 缺少必填参数 name（预期 400）
  4. 手机号格式错误（预期 400）
  5. 密码长度不足（预期 400）

要求：
  1. 使用 @pytest.mark.parametrize 做数据驱动
  2. 包含 conftest.py 的 fixture 配置（base_url, headers）
  3. 断言：状态码 + 响应体结构 + 业务码
  4. 测试间数据隔离（每个用例用独立手机号）
  5. 有清晰的中文注释

输出：完整的 test_user_register.py + conftest.py 代码

🔍 AI 生成脚本后的检查要点

  • 断言是否覆盖了所有关键字段
  • 测试数据是否能在不同环境运行
  • fixture 是否正确处理了 setup/teardown
  • 异常场景的断言逻辑是否正确
  • 是否有硬编码的环境地址或密钥

🐛 让 AI 帮你排查脚本报错

  • 把完整错误信息 + 相关脚本代码一起喂给 AI
  • 说明运行环境（Python 版本、依赖版本）
  • AI 通常能精确定位原因并给出修复方案
  • 连续报错时让 AI 整体分析而不是逐条修改
💻
🎯 实操任务（20 分钟）
为项目中一个真实的核心接口，用 AI 生成完整的 Pytest 自动化测试脚本，并在本地环境跑通至少一个测试用例。遇到报错时使用 AI 辅助调试。

Bug 管理与缺陷分析提效

⏱ 20 分钟

💡 高质量 Bug 报告的价值
高质量的 Bug 报告 = 更快的修复速度。很多 Bug 被开发"打回来"的原因是描述不清楚。用 AI 规范化 Bug 报告，可以显著减少沟通来回次数。

AI 辅助 Bug 报告规范化

💬 Bug 报告规范化提示词
基于以下测试过程描述，生成一份规范的 Bug 报告：
[口语化描述测试发现的问题]

示例描述："在手机号登录页面，输入正确账号密码后点登录，有时候会跳转失败，
           刷新后又好了，在 Chrome 上复现了 3 次"

输出格式：
- 标题：一句话概括问题（模块 + 现象）
- 严重级别：Critical/High/Medium/Low（附判断依据）
- 缺陷类型：功能缺陷/性能问题/UI 问题/安全漏洞
- 环境信息：浏览器/系统/版本号 [占位，请补充]
- 复现概率：必现/高频/偶现/极低（说明依据）
- 前置条件：
- 复现步骤：（编号步骤，每步一行）
- 实际结果：
- 预期结果：
- 可能原因分析（猜测，供开发参考）
- 附件说明：建议截图/录屏/日志的关键标注点
📌 标题
【登录模块】手机号密码登录偶发跳转失败，刷新后恢复正常
🔴 严重级别
High — 影响核心登录流程，偶发但用户无明显错误提示，可能导致用户流失
🔄 复现概率
偶现（3/10次，Chrome 环境下）
📋 复现步骤
1. 打开登录页
2. 输入已注册手机号 + 正确密码
3. 点击"立即登录"按钮
❌ 实际结果
页面无响应或停在登录页，无错误提示，无网络请求发出（偶发）
✅ 预期结果
登录成功，2 秒内跳转至首页，URL 变为 /home

AI 辅助根因分析 + 回归范围判断

🔍 根因分析提示词要素

  • 输入：错误日志 + 相关代码片段 + Bug 现象描述
  • AI 输出：可能根因（排序）+ 排查路径 + 修复方向建议
  • 价值：帮你在跟开发沟通时更有针对性，减少"你来看一下"式沟通
