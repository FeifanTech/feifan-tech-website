1b

反向链路：用测试视角审查需求

发现需求遗漏 = 测试用例覆盖了但 PRD 没有描述的场景

💬 双向审查分析提示词
以下是一个功能的相关材料：

【测试用例列表】
[粘贴测试用例表格或列表]

【对应的 PRD 章节】
[粘贴 PRD 内容]

请分析三个维度的完整性问题：

1. 测试用例 → PRD 缺失
   有哪些测试用例场景在 PRD 中找不到对应的需求描述？
   （说明 PRD 描述可能遗漏或不够清晰，附具体用例编号）

2. PRD → 测试用例缺失
   PRD 中有哪些功能描述没有对应的测试用例？
   （说明测试覆盖可能存在遗漏，附具体 PRD 章节引用）

3. AC 可测试性评估
   哪些 AC 描述模糊、难以直接转化为可执行的测试用例？
   给出具体的改写建议。

输出：三个问题分别列出，每条附上具体条目、问题描述和修改建议
💡 反向审查的真正价值
这个提示词的核心价值不是"找 Bug"，而是在功能上线前就发现需求文档与测试用例之间的不一致和遗漏，从根源上减少返工——这是 PM+QA 兼任角色独有的能力。
1c

全链路实操演示

复用前两期成果，现场走完完整链路

  1. PRD 功能描述 → AI 提取测试点 + 生成 AC

    直接使用第一期产出的 PRD 章节，复用第一期的 AC 生成提示词。

    📋 复用第一期成果
  2. AC → AI 生成测试用例矩阵

    直接使用第二期的"从 PRD + AC 直接生成测试用例矩阵"提示词，覆盖正常/异常/边界。

    🧪 复用第二期成果
  3. 测试用例矩阵 → AI 生成测试数据 JSON

    基于测试用例中的参数，批量生成覆盖所有场景的测试数据集（合法/边界/非法/业务异常）。

    🗂 数据驱动基础
  4. 测试用例 + 接口文档 → AI 生成 Pytest 脚本骨架

    以测试数据 JSON 作为 @pytest.mark.parametrize 的输入，生成数据驱动的完整测试脚本。

    ⚙️ 代码可直接运行
  5. 反向审查：用测试用例检查 PRD 完整性

    将步骤 2 生成的测试用例和步骤 1 的 PRD，一起喂给 AI 进行双向完整性审查。

    🔄 闭环质量保障
📦 全套产出物打包
完成以上 5 步后，你将拥有一个功能的完整质量文档包：PRD（含 AC）+ 测试用例矩阵 + 测试数据 JSON + Pytest 测试脚本 + 双向审查报告。这是课程的核心产出物。
💻
🎯 实操演示（20 分钟）
选取一个中等复杂度的真实功能（有 3-5 个 AC 的规模），现场完整走一遍 5 步链路。讲师同步演示，学员同步操作，遇到问题当场解决。
2

需求变更的联动管理

⏱ 20 分钟

⚠️ 需求变更是质量风险最高的时刻
需求变更后，测试用例同步更新是最容易被遗漏的事，尤其在 PM 和 QA 是同一个人、时间又紧的情况下。AI 辅助的变更影响分析，可以在 5 分钟内完成原本需要 1-2 小时的人工梳理。
💬 变更影响分析提示词
原始需求（变更前）：
[粘贴原始功能描述或 AC]

变更内容：
[描述具体变更，如："将最大购买数量从 99 改为 999"
 或"新增了'企业用户'角色，享有额外的折扣权限"]

历史测试用例（可选）：
[粘贴已有的相关测试用例]

请分析：
1. 受影响的原有测试用例（需要修改的，标注修改方向）
2. 新增了哪些需要补充的测试场景（直接给出用例草稿）
3. 是否影响其他关联功能（潜在的回归风险点）
4. 建议的回归测试优先级（High/Medium/Low）

输出：变更影响清单（用例ID + 影响类型：修改/删除/新增 + 具体说明）
2a

建立变更联动 SOP

在 Dify 中配置为自动化工作流，收到变更即自动触发分析

📋 变更联动 SOP 工作流
⚡ 触发条件：收到需求变更通知（飞书消息 / 邮件 / 项目管理系统变更）
1
AI 自动分析变更内容，提取变更范围和关键字段变化
2
检索知识库中的历史测试用例，生成受影响用例清单（修改/删除/新增）
3
基于变更内容，生成新增测试用例初稿（含测试数据建议）
4
输出变更评估报告（影响范围 + 回归建议 + 工作量估算）
5
发送飞书通知（包含摘要 + 完整报告链接），并更新项目管理系统

🔧 在 Dify 中配置

  • 创建工作流类型的 Workflow
  • 触发器：HTTP Webhook（项目管理系统调用）
  • Step 1-3：LLM 节点（使用变更影响分析提示词）
  • Step 4：RAG 检索节点（接入测试用例知识库）
  • Step 5：飞书通知节点

📋 配置要点

  • 测试用例知识库需要定期更新（建议每次回归后同步）
  • 变更通知需要包含足够上下文（原描述 + 变更后描述）
  • 输出格式固定，方便后续自动化处理
  • 人工最终确认：AI 的分析是建议，最终判断由人来做

3a

Agent 案例一：自动化测试报告生成

从原始测试结果到可发送的飞书报告，全程自动

A1
自动化测试报告生成 Agent
适用场景：每次 CI/CD 执行后自动生成和发送测试报告
📥 输入：测试执行结果（pytest XML 报告 / 截图 / 日志文件）
🤖 Agent 自主执行步骤
1
解析 pytest XML / JSON 报告，提取测试用例执行详情
2
统计通过率 / 失败率 / 跳过率，与上次执行对比变化趋势
3
对失败用例进行根因分类（断言失败 / 环境问题 / 超时 / 代码 Bug）
4
生成可视化测试报告（HTML 格式，含折线图和饼图）
5
发送飞书通知（摘要卡片 + 完整报告链接 + 失败用例列表）
🔧 实现方式（Dify）
Dify 工作流 → HTTP 触发器（CI/CD 回调）→ LLM 节点（分析报告）→ 代码节点（统计计算）→ 模板渲染（HTML 报告）→ 飞书 Bot 节点（发送通知）
3b

Agent 案例二：Bug 分类与路由

新 Bug 提交后自动分类、匹配历史缺陷、建议负责人

A2
Bug 分类与路由 Agent
适用场景：每次提交 Bug 后自动处理，减少人工分配时间
📥 输入：新提交的 Bug 描述（口语化或初步规范化的报告）
🤖 Agent 自主执行步骤
1
判断严重级别（Critical/High/Medium/Low）并给出判断依据
2
识别所属模块和功能（基于 Bug 描述的语义理解）
3
从历史 Bug 知识库检索相似缺陷（RAG 检索），判断是否为重复 Bug
4
根据模块归属和历史数据，建议最合适的处理人
5
通过 MCP 接口更新 Bug 跟踪系统，并发送飞书通知给相关人员
⚡ 关键依赖
历史 Bug 知识库（RAG）是这个 Agent 有效运作的核心。需要将历史 Bug 数据（至少 200+ 条）导入 Dify 知识库，按模块和类型分类标注。
3c

Agent 案例三：需求可测试性评估

新 PRD 提交后自动评估可测试性，生成测试准备清单

A3
需求可测试性评估 Agent
适用场景：需求评审前自动评估，提前发现需求质量问题
📥 输入：新版 PRD 文档（或 PRD 章节）
🤖 Agent 自主执行步骤
1
提取所有功能点（解析 PRD 结构，按模块分类）
2
评估每个功能点的可测试性（是否有明确的验收标准？边界是否清晰？）
3
为可测试的功能点自动生成测试用例初稿（用于测试准备）
4
对不可测试的功能点生成澄清问题列表（发送给 PM 确认）
5
输出可测试性评估报告（评分 + 测试准备状态 + 风险提示）
💻
🎯 实操：在 Dify 中搭建需求可测试性评估 Agent（25 分钟）
步骤： 1. 在 Dify 创建 Agent 类型应用
2. 配置系统提示词（可测试性评估专家角色）
3. 接入公司产品决策库知识库（RAG）
4. 配置工具：文档解析 + 知识库检索
5. 测试：上传一份真实 PRD，验证评估结果

构建团队级 PM×QA 知识库与 Skills

⏱ 25 分钟

PM+QA 兼任角色积累了两种最有价值的知识：产品决策记录（为什么做这个功能）和质量经验记录（这个地方容易出 Bug）。

📋 产品决策库
  • 历史 PRD 文档
  • 需求变更记录与原因
  • 设计决策文档
  • 需求评审纪要
新需求可检索历史相关决策，避免重复踩坑
🧪 测试经验库
  • 历史 Bug 按模块归类
  • 高风险模块的测试重点
  • 测试用例模板库
  • 自动化脚本复用库
新功能测试时 AI 自动推荐历史关注点
✅ 验收标准库
  • 各模块的验收条件规范
  • 典型 AC 示例（按业务类型）
  • 行业通用测试标准
  • 性能基线数据
AI 写 AC 时有公司规范参考，输出更一致

30 天个人行动计划

🗓 建议的 30 天落地节奏
第 1 周
巩固基础：在当前 1 个真实项目中，完整跑通需求→AC→测试用例的正向链路，产出一份质量文档包。
第 2 周
提升自动化：为最核心的 3-5 个接口生成 AI 辅助的 Pytest 脚本，实现本地可运行的自动化套件。
第 3 周
Skills 上线：在 Dify 中配置并上线 AC 生成器和 Bug 报告规范器两个 Skill，在日常工作中正式使用。
第 4 周
沉淀分享：整理 30 天内效果最好的 3 个提示词/工作流，在团队分享会中分享，沉淀到团队知识库。
