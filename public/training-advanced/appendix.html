二

本课程专属工具清单

按使用场景和技术门槛整理，方便按需选用

🤖
Claude（长上下文）
技术门槛：低
支持超长上下文（200K token），可以处理完整 PRD 文档、大量测试用例、完整代码库。
主要用途：长 PRD 分析、完整测试用例生成、复杂提示词执行、代码审查
推荐场景：处理超过 3000 字的文档；需要多步骤推理的任务
🌐
Gemini（联网版）
技术门槛：低
支持实时联网搜索，可以获取最新的竞品信息、行业动态、产品发布说明。
主要用途：竞品信息收集、实时市场分析、最新技术文档查阅
推荐场景：竞品分析第一步（信息收集）；了解竞品最新功能更新
⚙️
Dify
技术门槛：中
低代码 AI 应用开发平台，支持 Skills 配置、RAG 知识库、工作流和 Agent 搭建。
主要用途：Skills 配置、RAG 知识库建设、变更联动工作流、Agent 搭建
学习曲线：前置课程第四期已有基础，本课程直接进阶应用
💻
Cursor
技术门槛：中
AI 原生代码编辑器，支持自然语言生成/修改代码，内置 AI 调试和代码解释功能。
主要用途：测试脚本生成与调试、代码重构、脚本维护
推荐场景：生成 Pytest/Playwright 脚本后的调试阶段
🎭
Playwright
技术门槛：中
现代 UI 自动化测试框架，支持 Python/TypeScript，有强大的等待机制和调试工具。
主要用途：UI 端到端自动化测试、页面录制转脚本
特别优势：Playwright Codegen 可以录制操作生成脚本骨架
🧪
Pytest
技术门槛：中
Python 主流测试框架，支持参数化、fixture、丰富的插件生态（pytest-html/allure）。
主要用途：接口自动化测试、数据驱动测试
配合工具：requests（HTTP）/ allure（报告）/ pytest-xdist（并行）
🦗
Locust
技术门槛：中
Python 编写的分布式性能测试工具，支持真实用户行为模拟，有实时监控 Web 界面。
主要用途：接口性能测试、并发场景验证
适用场景：核心接口上线前的性能基准测试
📄
Swagger / OpenAPI
技术门槛：低
接口文档标准格式，作为 AI 生成测试数据和测试脚本的标准化输入源。
主要用途：接口测试数据生成的输入文档、Pytest 脚本生成的参数来源
使用方式：直接粘贴 Swagger YAML/JSON 到提示词中

3a

完整提示词模板

直接复制使用，替换 [ ] 内容即可

P1 · 需求侧
需求深挖框架
所在位置：第一期 · 模块 2 · 需求发现与澄清
Claude
角色：你是一位经验丰富的产品经理，擅长用结构化方法澄清模糊需求

背景：[粘贴客户的原始需求描述或会议记录]

任务：
  1. 识别需求中的模糊表述，列出需要澄清的问题（按优先级排序）
  2. 识别客户可能没有明说的隐性需求
  3. 识别潜在的边界冲突（与现有功能 / 其他需求的矛盾点）
  4. 给出建议的确认问题清单（可直接发给客户）

格式：分四个部分输出，每部分不超过 5 条，使用 Markdown 格式
💡 使用技巧：第 4 部分的"确认问题清单"可以直接复制发给客户/业务方，语气已经默认较为礼貌。如果需要更简短的版本，在提示词末尾加"每部分压缩到 3 条以内"。
P2 · 需求侧
PRD 功能规范化
所在位置：第一期 · 模块 3 · PRD 写作全流程
Claude
以下是一段口语化的功能描述：
[粘贴功能描述，如"用户可以搜索商品"]

请将其转化为规范的产品文档语言，包含以下部分：
1. 正常流程（用户操作步骤 + 系统响应，尽量具体）
2. 边界条件（输入限制、数量限制、长度限制等）
3. 异常流程（错误处理、降级策略、友好提示）
4. 性能要求（响应时间、并发要求，如无法推断请给出建议值）
5. 数据约束（字段格式、长度、精度限制）

如果以上某类信息在原描述中无法推断，请明确标注"【待确认】"并给出建议值供参考。
输出：规范化后的功能说明（Markdown 格式）
💡 使用技巧：处理之后，重点审查 AI 标注的"【待确认】"项目——这些是你在需求评审前必须确认的点。
P3 · 需求侧
AC 一体化生成
所在位置：第一期 · 模块 3 · AC 一体化生成
Claude
基于以上功能描述（[粘贴功能描述或在对话中直接引用上文]），
生成完整的验收条件（Acceptance Criteria）

格式要求：
- 使用 Given-When-Then 结构（三段式，清晰明确）
- 必须覆盖：正常流程 / 边界情况 / 异常场景 / 性能场景（如适用）
- 每个场景单独一行

输出格式：Markdown 表格
列：场景描述 | Given（前置条件）| When（触发动作）| Then（预期结果）| 优先级（P0/P1/P2）

优先级定义：
- P0：核心流程，任何情况下不可缺少
- P1：重要场景，正式上线前必须覆盖
- P2：边缘场景，迭代中可逐步覆盖
💡 使用技巧：这个模板和 P5（测试用例矩阵生成）是"接力"关系——P3 的输出直接作为 P5 的输入。完成 AC 后立刻保存，第二期直接复用。
P5 · 测试侧
测试用例矩阵生成（含反向需求审查）
所在位置：第二期 · 模块 2 · 从 PRD 直生测试矩阵
Claude
角色：你是一位同时具备产品和测试思维的高级测试工程师

输入 1 - PRD 功能描述：
[粘贴功能描述]

输入 2 - 验收条件（AC）：
[粘贴 Given-When-Then 格式的 AC 表格]

任务：
  1. 提取所有可测试点（每个 AC 必须对应至少 1 个测试用例，无对应请说明）
  2. 为每个测试点生成测试用例（正常流 + 异常流 + 边界场景）
  3. 识别高风险区域，标注需要重点测试的场景（说明风险原因）
  4. 指出 PRD/AC 中描述不清晰、无法测试的地方（反向审查）

输出格式：
  A. 测试用例表格：
     用例ID | 模块 | 场景描述 | 前置条件 | 操作步骤（编号）| 预期结果 | 优先级 | 对应AC编号
  B. 风险标注：[高风险场景 + 风险原因 + 测试建议]
  C. 需求疑问：[模糊/无法测试的 PRD/AC 条目 + 具体问题 + 修改建议]
💡 使用技巧：输出 C 部分（需求疑问）往往比 A 部分（测试用例）更有价值——这是需求文档质量问题的精准定位，可以直接作为需求评审的补充意见反馈给 PM（就是自己）。
P6 · 测试侧
测试数据批量生成
所在位置：第二期 · 模块 3 · 测试数据生成
Claude
生成 [N] 条 [功能名称] 测试数据，要求包含：

- [n1] 条完全合法的数据（覆盖各种典型用户类型/使用场景）
- [n2] 条边界值数据（[指定边界，如姓名最大长度/数值边界]）
- [n3] 条非法数据（SQL 注入/XSS 脚本注入/特殊字符/超长输入/类型错误）
- [n4] 条业务异常数据（[指定业务异常，如重复注册/黑名单/已过期]）

字段：[列出所有字段名和类型，如 name(string), age(int), email(string)]

输出格式：JSON 数组
每条数据额外包含：
  - "testType": "valid" | "boundary" | "invalid" | "bizError"
  - "testScenario": "说明这条数据测试的具体场景"
  - "expectedResult": "预期的系统响应（成功/具体错误信息）"
💡 使用技巧：生成的 JSON 可以直接用于 Pytest 的 @pytest.mark.parametrize，结合 P7（Pytest 脚本生成）形成完整的数据驱动测试套件。
P7 · 测试侧
Pytest 自动化测试脚本生成
所在位置：第二期 · 模块 4 · 自动化脚本生成
CursorClaude
角色：你是一位精通 Python 自动化测试的工程师

接口文档：
  URL: [HTTP方法] [接口路径，如 POST /api/v1/users/register]
  请求头：[Authorization / Content-Type 等]
  请求体：[JSON 格式的参数结构和说明]
  成功响应：[HTTP 200 + 响应体结构]
  错误响应：[各错误码 + 含义，如 400/409/500]

测试场景（来自测试用例矩阵）：
  [列出需要覆盖的测试场景，每行一个]

代码要求：
  1. 使用 @pytest.mark.parametrize 做数据驱动（测试数据 JSON 作为输入）
  2. 包含 conftest.py：
     - base_url fixture（支持从环境变量读取，默认值为开发环境地址）
     - auth_headers fixture（自动获取测试账号 token）
  3. 断言覆盖：HTTP 状态码 + 响应体关键字段 + 业务状态码
  4. 测试间数据隔离（如用唯一 phone/email 避免冲突）
  5. 清晰的中文注释（说明每个测试的目的）

输出：
  1. test_[功能名].py（完整测试文件）
  2. conftest.py（fixture 配置文件）
  3. 简短的运行说明（pip install 哪些依赖 + 运行命令）
💡 使用技巧：优先使用 Cursor 来执行这个提示词（可以在同一工具中直接运行和调试）。生成后检查 conftest.py 中的环境变量名称是否与团队约定一致。

Bug 报告规范化
所在位置：第二期 · 模块 5 · Bug 管理与缺陷分析
Claude
基于以下测试过程描述，生成一份规范的 Bug 报告：
[粘贴口语化的问题描述，越详细越好]

输出格式（严格按此格式输出，不要省略任何字段）：

**环境信息**：
- 浏览器/客户端版本：[请填写]

**预期结果**：[按照需求规格，应该发生什么]

**可能原因**（供开发参考）：[基于描述的合理猜测]

**附件说明**：[建议截图/录屏/日志应该标注哪些关键点]
💡 使用技巧：在 Dify 中将此提示词配置为 Bug 报告规范器 Skill，接入飞书机器人后，测试人员直接在飞书对话框描述问题即可自动生成规范报告。
P9 · 打通层
变更影响分析
所在位置：第三期 · 模块 2 · 需求变更联动管理
Claude
原始需求（变更前）：
[粘贴原始功能描述或 AC 列表]

变更内容（变更后）：
[具体描述变更，如新增字段/修改规则/删除功能/性能要求变化]

历史测试用例（如有）：
[粘贴相关测试用例，或说明用例编号范围]

请分析：

1. 受影响的原有测试用例（需要修改的）
   格式：用例ID | 影响类型（修改/删除）| 需要如何改变

2. 需要新增的测试场景（因变更引入的新测试点）
   格式：直接给出测试用例草稿（含前置条件 + 操作步骤 + 预期结果）

3. 关联功能回归风险点
   格式：关联功能 | 风险描述 | 建议验证方式

4. 回归测试优先级建议
   High：必须在发布前验证
   Medium：建议在发布前验证
   Low：可以在下个迭代验证

输出：变更影响评估报告（Markdown 格式）
💡 使用技巧：配置为 Dify 工作流后，收到变更通知时自动触发，5 分钟内输出影响评估报告，确保变更不遗漏测试同步。
P10 · 打通层
双向审查分析（需求 ↔ 测试完整性检验）
所在位置：第三期 · 模块 1 · 需求与测试双向打通
Claude
以下是同一个功能的 PRD 和测试用例：

【测试用例列表】
[粘贴测试用例，至少包含：用例ID / 场景描述 / 优先级]

【PRD 功能描述 + AC】
[粘贴 PRD 功能描述和 Given-When-Then 格式的 AC]

请从三个维度进行双向完整性审查：

维度 1：测试 → PRD（测试视角发现的需求问题）
  有哪些测试用例场景在 PRD/AC 中找不到对应的需求描述？
  （意味着：PRD 可能遗漏了描述，或 AC 不够完整）
  格式：用例ID | 场景描述 | PRD 中缺失的内容 | 建议补充

维度 2：PRD → 测试（需求视角发现的测试盲区）
  PRD/AC 中有哪些功能描述没有对应的测试用例？
  （意味着：测试覆盖可能存在遗漏）
  格式：PRD 章节 / AC 条目 | 未覆盖的场景 | 建议新增用例

维度 3：AC 可测试性评估
  哪些 AC 描述模糊、难以直接转化为可执行的测试用例？
  格式：AC 条目 | 模糊点说明 | 改写建议

总结：整体完整性评分（1-10分）+ 最优先处理的 3 个问题
💡 使用技巧：这是三期课程中最有价值的提示词之一。建议每次需求评审前都运行一次，发现问题越早修复成本越低。维度 1 的结果可以直接用于需求评审会议的"测试评审意见"部分。

📌 评分后的建议行动
  • 1-2 分：重新学习对应模块，找一个真实任务练习
  • 3 分：已经会用但不稳定，需要 2-3 次真实项目强化
  • 4 分：日常稳定使用，可以尝试优化提示词或分享给团队
  • 5 分：已经内化，可以帮助他人，考虑沉淀为团队 SOP
🎯 找出你的 2 个优先练习方向
  • 选出得分最低的 2-3 个维度
  • 每个方向制定一个"30 天内在真实项目中使用 3 次"的目标
  • 每次使用后记录：用了什么提示词 / 输出是否满足期望 / 如何优化
  • 第 30 天重新填写此自评表，看进步幅度
